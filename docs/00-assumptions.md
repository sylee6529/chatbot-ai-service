# 00 — 설계 결정 사항

AI 챗봇 서비스 구현 전에 확정한 도메인, API, 데이터 저장 방식의 기준을 기록합니다.
필수 기능은 간단하고 일관된 계약으로 구현하고, 운영 편의나 확장 기능은 별도 범위로
분리합니다.

---

## A. 사용자 & 인증

| # | 결정 | 설명 |
|---|------|------|
| A1 | 사용자 필드: **`email`**(신원, unique, 소문자 정규화), **`password`**, **`name`**(이름), **`created_at`**, **`role`**. | 공개 계약을 단순하게 유지한다. `username`/`displayName` 같은 별도 표시명 필드는 두지 않는다. |
| A2 | 로그인 식별자는 `email`. 이메일 **형식 검증**(RFC 기본 패턴) 적용. | 회원가입과 로그인 모두 이메일을 기준으로 사용자를 찾는다. |
| A3 | 로그인 성공 시 **access token(JWT)만** 발급. refresh token은 현재 구현 범위 밖이며 향후 개선 목록에 기재. | 인증 흐름을 access token 중심으로 단순화한다. 토큰 갱신/폐기는 별도 기능으로 확장한다. |
| A4 | JWT는 **무상태(stateless)**. 클레임: `sub`(사용자 id), `email`, `role`, `iat`, `exp`. 기본 TTL **1시간**(환경변수 설정 가능). | 요청마다 DB 조회 없이 인가하기에 충분. |
| A5 | 비밀번호: **BCrypt** 해시로만 저장. 최소 8자 + 영문/숫자 포함 기본 강도 검증. | 평문 저장을 금지하고 기본적인 가입 품질을 보장한다. |
| A6 | **초기 관리자**는 환경변수(`ADMIN_EMAIL`, `ADMIN_PASSWORD`, `ADMIN_NAME`)로 기동 시 없으면 시딩. README에 문서화. | API를 통한 자가 승격을 막고, 배포 환경에서 관리자 계정을 명확히 주입한다. |

## B. 역할 & 인가

| # | 결정 | 설명 |
|---|------|------|
| B1 | 역할: **`MEMBER`**(가입 시 기본) / **`ADMIN`**. | 두 역할만 사용해 인가 규칙을 명확히 유지한다. |
| B2 | 인가 검사는 **서비스 계층**에서 강제. 소유권 위반 → `403`, 없는 리소스 → `404`. | 컨트롤러 진입 이후에도 소유권과 역할을 반드시 확인한다. |

## C. 스레드 & 30분 규칙

| # | 결정 | 설명 |
|---|------|------|
| C1 | "30분 이내"는 사용자의 **가장 최근 채팅 `created_at`** 기준. 경계 포함(≤ 30:00 재사용, > 30:00 신규). | 경계값까지는 같은 대화 흐름으로 보고 기존 스레드를 재사용한다. |
| C2 | 첫 질문이거나 이전 채팅이 없으면 **새 스레드** 생성. | 사용자의 첫 상호작용은 항상 새 스레드의 시작점이다. |
| C3 | 30분 기준은 사용자 전체에서 **가장 최근 채팅 1건**. 이 채팅이 속한 스레드가 "existing latest thread". | 사용자는 한 번에 하나의 최신 대화 흐름을 이어가는 것으로 처리한다. |
| C4 | 30분 값은 설정 가능(`chat.thread.window-minutes=30`). | 코드 변경 없이 조정 가능. |
| C5 | 채팅 생성은 사용자 단위 직렬화로 처리한다. 트랜잭션 안에서 `users` 행을 `SELECT ... FOR UPDATE`로 잠근 뒤 최신 채팅 조회, 스레드 결정, 채팅 저장을 수행한다. | 같은 사용자가 동시에 질문을 보내도 30분 규칙이 중복 스레드를 만들지 않도록 한다. PostgreSQL과 H2 테스트 모두에서 사용할 수 있는 row lock을 우선한다. |

## D. 채팅 & AI 제공자

| # | 결정 | 설명 |
|---|------|------|
| D1 | 채팅 행: **`question`**, **`answer`**, **`created_at`**, 스레드 참조. `model`도 함께 저장. | 질문/답변 한 쌍을 하나의 채팅으로 저장하고, 사용된 모델을 추적한다. |
| D2 | AI 컨텍스트 = 같은 스레드의 이전 채팅들(오래된→최신 순). | 모델이 이전 흐름을 이해할 수 있도록 같은 스레드의 기록만 전달한다. |
| D3 | 현재 구현은 **OpenAI-compatible API** 1개를 기본 provider로 사용한다. `AI_API_KEY`, `AI_BASE_URL`, `AI_DEFAULT_MODEL` 환경변수로 설정. | provider별 자동 선택은 하지 않되, `AiClient` 경계로 향후 provider adapter를 추가할 수 있게 한다. |
| D4 | 요청 파라미터 `model`이 있으면 `AI_DEFAULT_MODEL` 오버라이드. 사용된 모델을 `model` 컬럼에 저장. | `model`은 현재 provider에 전달할 모델명이다. 모델명만으로 provider를 자동 선택하지 않는다. |
| D5 | `model`은 빈 값/길이/기본 문자셋만 애플리케이션에서 검증하고, 실제 존재 여부는 provider 응답에 맡긴다. | provider 호출 전에는 모델 카탈로그를 확정할 수 없으므로 모델 미존재/미지원은 `502`로 매핑한다. |

## E. 피드백

| # | 결정 | 설명 |
|---|------|------|
| E1 | 피드백 필드: **`user_id`**, **`chat_id`**, **`is_positive`(boolean)**, **`created_at`**, **`status`**. | 피드백은 긍정/부정과 처리 상태만 저장한다. 자유 텍스트 코멘트는 두지 않는다. |
| E2 | `is_positive`: `true`(긍정) / `false`(부정). 필터는 이 boolean으로 수행. | API와 DB 모두 boolean으로 표현해 필터링 기준을 단순하게 유지한다. |
| E3 | `status` ∈ `{pending, resolved}`, 기본 `pending`. **관리자만** 변경. | 사용자는 피드백을 생성하고, 관리자는 처리 상태를 관리한다. |
| E4 | 유일성: **`(user_id, chat_id)`** — DB 유니크 제약 + 서비스 검사 → `409`. | 동일 사용자가 같은 채팅에 중복 피드백을 남기지 못하게 한다. |
| E5 | 멤버는 **본인 채팅**에만, 관리자는 **모든 채팅**에 피드백 생성 가능. | 소유권 기반 인가와 관리자 예외를 동일하게 적용한다. |

## F. 분석 & 리포팅

| # | 결정 | 설명 |
|---|------|------|
| F1 | "하루" = 요청 시각 기준 **롤링 24시간**(`now() - 24h` ~ `now()`). 달력일 아님. | 호출 시점에 따라 일관된 24시간 범위를 계산한다. |
| F2 | 집계 항목: **회원가입 수**, **성공 로그인 수**, **대화 생성 수**. `activity_logs` 테이블에 `SIGNUP`, `LOGIN`, `CHAT_CREATED` 유형으로 영속화. | 로그인 집계는 토큰 발급에 성공한 로그인만 센다. 실패 로그인은 현재 집계 대상이 아니다. |
| F3 | 활동 로그는 핵심 작업이 성공한 뒤 best-effort로 기록한다. 로그 저장 실패는 경고 로그만 남기고 회원가입/로그인/채팅 생성을 롤백하지 않는다. | 분석 보조 데이터 실패가 핵심 사용자 기능을 실패시키지 않게 한다. 중복/누락 가능성은 운영 로그로 추적한다. |
| F4 | CSV 리포트: 최근 24시간 **모든 사용자의 대화 목록 + 어떤 사용자가 생성했는지** 포함. `Content-Type: text/csv`. | 컬럼은 `chat_id`, `thread_id`, `user_id`, `user_email`, `user_name`, `model`, `question`, `answer`, `chat_created_at`으로 고정한다. |
| F5 | 두 분석 기능 모두 **관리자 전용** (`403` for member). | 전체 사용자 데이터가 포함되므로 관리자 권한으로 제한한다. |

## H. 영속화 & 플랫폼

| # | 결정 | 설명 |
|---|------|------|
| H1 | **Flyway** 마이그레이션. 앱은 `ddl-auto=validate`. 테스트는 H2(PostgreSQL 호환 모드)에서 Flyway 실행. | `./gradlew test`가 Docker 없이 통과해야 함. |
| H2 | 타임스탬프: `Instant`(UTC) → `timestamptz`. | 시간대 오해 없이 저장하고 ISO-8601 UTC로 응답한다. |
| H3 | PK: **BIGINT identity**. UUID는 향후 옵션. | 간단한 FK 조인과 디버깅 편의를 우선한다. |
| H4 | 애플리케이션 삭제는 **소프트 삭제**로 처리한다. 도메인 테이블은 `deleted_at`을 두고, 삭제된 데이터는 일반 조회/컨텍스트/리포트에서 제외한다. | 복구 가능성과 감사 가능성을 남기면서 사용자 조회에서는 보이지 않게 한다. |
| H5 | 유니크 제약은 soft-deleted row도 포함한다. `users.email`, `feedbacks(user_id, chat_id)`는 `deleted_at` 여부와 무관하게 재사용하지 않는다. | PostgreSQL partial unique index와 H2 호환성 차이를 피하고, 삭제 후 같은 키 재사용 정책을 단순하게 유지한다. |
| H6 | 로컬 DB: `aichatbot`, user `postgres`. 비밀번호는 환경변수. | 비밀 정보 미커밋. |

| # | 결정 | 설명 |
|---|------|------|
| I1 | 모든 엔드포인트: **`/api/v1`** 접두사. | 버전 관리. |
| I2 | 정상 응답은 도메인 객체를 바로 반환하고, 목록 응답만 pagination wrapper를 사용한다. | 불필요한 중첩을 피하고 목록 메타데이터만 표준화한다. |
| I3 | 에러 응답: `{ "timestamp", "status", "error", "message", "path" }` 일관 적용. 필드별 오류 배열은 두지 않는다. | 예측 가능한 클라이언트 처리. |
| I4 | 페이지네이션: `page`(0-base), `size`(1~100), `sort`. 응답: `content`, `page`, `size`, `totalElements`, `totalPages`. | 목록 응답 형식을 통일해 클라이언트 구현을 단순화한다. |
| I5 | AI provider 호출 실패는 `502`, 시간 초과는 `504`로 응답한다. 스트리밍 시작 후 실패는 SSE `error` 이벤트로 전달한다. | 외부 provider 장애와 일반 요청 검증 오류를 분리한다. |
| I6 | `/api/v1/health`는 공개하고, Swagger/OpenAPI 문서는 JWT 인증을 요구한다. | 상태 확인은 열어두되 API 문서는 보호 엔드포인트와 같은 정책을 적용한다. |
| I7 | 시각 표현: **ISO-8601 UTC** 문자열. | 클라이언트 간 명확. |

## 확정 결정 요약

- **AI provider** — OpenAI-compatible API 구현만 제공한다. 멀티 provider와 테스트용 가짜 클라이언트는 현재 구현 대상이 아니다.
- **삭제 정책** — 도메인 테이블의 `deleted_at` 기반 소프트 삭제로 통일한다. 삭제된 데이터는 일반 조회, AI 컨텍스트, CSV 리포트에서 제외한다.
- **유니크 제약** — soft-deleted row의 unique key도 재사용하지 않는다. H2 호환성을 위해 partial unique index에 의존하지 않는다.
- **30분 동시성** — 채팅 생성 시 사용자 row lock으로 사용자별 요청을 직렬화한다.
- **CSV 리포트** — 컬럼은 `chat_id`, `thread_id`, `user_id`, `user_email`, `user_name`, `model`, `question`, `answer`, `chat_created_at`으로 고정한다.
